# 執筆前の事前確認（preflight）

**いきなり書き始めない。** 記事の方向性が定まらないまま書くと、後から大きな手戻りになる。
まず以下を確認し、不明な点は簡潔な質問でユーザーにそろえてもらってから執筆に入る。

## 確認する項目

1. **目的・ゴール**：この記事で読者に何を持ち帰ってほしいか（学び / 手順の再現 / 意見の共有 など）。
2. **想定読者**：前提知識のレベル。専門用語をどこまで噛み砕くかが変わる。
3. **スタイル / トーン**：
   - 参加レポート / 技法・概念の解説 / ハンズオン・チュートリアル / 意見・ポエム など
   - 文体（です・ます / だ・である）。既存記事の文体に合わせるのが基本。
4. **フロントマター方針**：
   - `type`：`tech`（技術記事）か `idea`（アイデア・レポート等）か
   - `emoji`：記事を表す1つ
   - `topics`：1〜5個。記事の主題を反映しているか
   - `published`：新規は **必ず `false`**
5. **slug**：意味のある名前にするか、CLI のランダム生成でよいか（`zenn-article` スキル参照）。
6. **構成案**：見出しレベルの章立てを先に提示し、合意を取ってから本文を書く。

## 進め方

- 既に情報が十分そろっていれば、確認を最小限にしてすぐ構成案の提示に進む。
- 曖昧な点が複数あるときは、選択式の質問でまとめて確認し、往復回数を減らす。
- 参考資料（イベントページ・元ネタ）が与えられたら、事実を取り違えないよう先に目を通す。
- 確認が済んだら、合意した構成案にそって執筆し、`zenn-article` スキルで作成・プレビューする。
